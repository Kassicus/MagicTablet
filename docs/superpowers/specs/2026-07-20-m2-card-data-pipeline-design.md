# M2 — Card Data Pipeline — Design Spec

**Date:** 2026-07-20
**Milestone:** M2 (DESIGN.md §7, §8)
**Status:** Approved scope, pending spec review

> Master brief: `DESIGN.md` §7. M2 is a **build-time tool**, not app code: a standalone Python
> script that turns Scryfall bulk exports into a trimmed, FTS-indexed SQLite bundled in `assets/`.
> M3 wires Room to read it. Builds on M0/M1 (on `main`) but touches no app source except adding the
> generated DB under `app/src/main/assets/` (gitignored).

---

## 1. Purpose

Produce `app/src/main/assets/cards.db`: a compact, offline, full-text-searchable SQLite of Magic
cards + rulings (text only, no images/prices), from Scryfall's **Oracle Cards** and **Rulings** bulk
exports — so the app (M3+) can look up any card's oracle text and rulings fully offline.

## 2. Scope

**In scope (M2 ships):**
- `tools/build_card_db/build_card_db.py` — fetches Scryfall bulk data, streams + trims + filters it,
  writes the SQLite with an FTS5 index.
- `tools/build_card_db/requirements.txt` (ijson), `README.md`, and a unit-test file.
- `.gitignore` entries for the generated DB, the download cache, and the venv.
- The generated `app/src/main/assets/cards.db` (produced locally; **gitignored**, regenerated by the script).

**Out of scope (deferred):**
- Any app/Room wiring or card-lookup UI (**M3**).
- In-app WiFi sync (refresh cadence for M2 is: **re-run the script** — DESIGN.md §9's "manual re-bundle").
- Stamping the Room `room_master_table` identity hash into the DB (**M3** — see §9).
- Images, prices, non-oracle data (deliberately dropped).

## 3. Environment / tooling (verified 2026-07-20)

- **Python 3.12.3**, **pip 24.0** on the build Mac. `sqlite3` module → SQLite **3.45.1 with FTS5 working**.
- **Dependencies:** `ijson` (streaming JSON parse) via pip in a venv; `urllib` (download) + `sqlite3` +
  `json` from the stdlib. No `requests`.
- The script runs on the **Mac** (not the tablet). Network is required (Scryfall). Runs need to bypass
  the Bash sandbox for network (`dangerouslyDisableSandbox: true`).

## 4. Layout & outputs

```
tools/build_card_db/
  build_card_db.py        # main script (importable pure fns + main())
  test_build_card_db.py   # unittest for pure fns (no network)
  requirements.txt        # ijson
  README.md               # how to run (venv + install + run)
  .cache/                 # downloaded oracle_cards.json / rulings.json (gitignored)
  .venv/                  # (gitignored)
app/src/main/assets/
  cards.db                # generated output (gitignored)
```

`.gitignore` adds: `/app/src/main/assets/cards.db`, `tools/build_card_db/.cache/`,
`tools/build_card_db/.venv/`.

## 5. Pipeline

1. **Discover bulk URLs:** `GET https://api.scryfall.com/bulk-data` with headers
   `User-Agent: MagicTablet/0.1 (personal fan project)` and `Accept: application/json` (Scryfall API
   etiquette). From the returned list, take the `download_uri` of the items with `type == "oracle_cards"`
   and `type == "rulings"`.
2. **Download** each `download_uri` (a single large JSON **array** file — current Scryfall bulk is `.json`,
   not `jsonl.gz` as DESIGN.md §7 loosely says) to `.cache/`, streaming to disk. Reuse the cached file
   unless `--force` is passed.
3. **Parse + trim + filter** by streaming array elements with `ijson.items(f, "item")`:
   - **Exclude** a card if `layout` ∈ {`token`, `double_faced_token`, `emblem`, `art_series`} OR
     `name` starts with `"A-"` (Alchemy rebalance) OR it has no `oracle_id`.
   - **Extract** kept cards into the `Card` row (see §6). Collect kept `oracle_id`s into a set.
   - Stream rulings; keep only those whose `oracle_id` is in the kept set; write `Ruling` rows.
4. **Write SQLite** to a temp file, populate `Card` + `CardFts` + `Ruling`, create the index, `VACUUM`,
   then atomically move to `app/src/main/assets/cards.db`. Print stats (cards, rulings, bytes).

## 6. Schema & field mapping

Column names are camelCase to align with the future Room entities (M3), matching DESIGN.md §4.

```sql
CREATE TABLE Card (
  oracleId  TEXT PRIMARY KEY,
  name      TEXT NOT NULL,
  manaCost  TEXT,
  typeLine  TEXT,
  oracleText TEXT,
  keywords  TEXT          -- JSON array string
);
CREATE TABLE Ruling (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  oracleId   TEXT NOT NULL,
  publishedAt TEXT,
  text       TEXT
);
CREATE INDEX idx_ruling_oracleId ON Ruling(oracleId);
CREATE VIRTUAL TABLE CardFts USING fts5(
  name, oracleText, oracleId UNINDEXED,
  tokenize = 'unicode61 remove_diacritics 2'
);
```

**Card field mapping (Scryfall → column):**
- `oracleId` ← `oracle_id`; `name` ← `name`; `keywords` ← `json.dumps(keywords or [])`.
- `manaCost` ← `mana_cost`; if empty/absent and `card_faces` present → join non-empty face `mana_cost` with `" // "`.
- `typeLine` ← `type_line`; if absent and `card_faces` present → join face `type_line` with `" // "`.
- `oracleText` ← `oracle_text`; if absent and `card_faces` present → join face `oracle_text` with `"\n//\n"`.
  (Handles DFC / split / adventure / modal cards where text lives on faces.)

**Ruling field mapping:** `oracleId` ← `oracle_id`; `publishedAt` ← `published_at`; `text` ← `comment`.

**CardFts** is populated in lock-step with `Card` (same `name`, `oracleText`, `oracleId`). App search
(M3) runs `SELECT oracleId FROM CardFts WHERE CardFts MATCH ?` then joins `Card`.

## 7. Code structure (for testability)

`build_card_db.py` exposes pure, importable functions so logic is unit-tested without network/DB:
- `should_include(card: dict) -> bool` — the §5.3 exclusion rules.
- `extract_card(card: dict) -> dict` — the §6 field mapping (incl. `card_faces` fallback).
- `extract_ruling(ruling: dict) -> dict`.
- `main()` orchestrates fetch → download → build; guarded by `if __name__ == "__main__"`.

CLI: `python build_card_db.py [--force] [--out PATH]` (default out `app/src/main/assets/cards.db`).

## 8. Testing

- **Unit tests** (`test_build_card_db.py`, stdlib `unittest`, no network/DB): `should_include` excludes
  each excluded layout + an `A-` name + a no-oracle_id card, and includes a normal card; `extract_card`
  maps a simple card, and joins a `card_faces` DFC (name kept, mana/type/text joined) and one with a
  missing `oracle_text`; `extract_ruling` maps `comment` → `text`.
- **End-to-end run** (real network, on the Mac — the "device" equivalent): run the script, then validate
  the DB via `sqlite3`:
  - `Card` count > 0 and `Ruling` count > 0.
  - A known card is present: `SELECT oracleText FROM Card WHERE name = 'Lightning Bolt'` returns text.
  - FTS works: `SELECT name FROM CardFts WHERE CardFts MATCH 'lightning bolt'` includes it.
  - No excluded rows leaked: no `Card` name starts with `A-`.

## 9. Definition of done (M2)

- [ ] `tools/build_card_db/{build_card_db.py, requirements.txt, README.md, test_build_card_db.py}` exist.
- [ ] `.gitignore` excludes the generated DB, `.cache/`, `.venv/`.
- [ ] Unit tests pass (`python -m unittest`).
- [ ] A real run produces `app/src/main/assets/cards.db` with populated `Card` / `Ruling` / `CardFts`.
- [ ] Validation queries pass: Lightning Bolt present with oracle text; FTS `MATCH` finds it; no `A-` names.
- [ ] Stats (card/ruling counts, DB size) printed on completion.
- [ ] `README.md` credits the data source (Scryfall) and notes card names/oracle text are Wizards of the Coast IP shown for reference only (DESIGN.md §11).

## 10. M2 → M3 seam (noted, not built here)

- M3 defines Room `Card`/`Ruling` entities + a DAO with a raw `MATCH` query over `CardFts`, opened via
  `Room.databaseBuilder(...).createFromAsset("cards.db")`. Room validates the asset against a
  `room_master_table` identity hash; the plain DB M2 builds has none. M3 will either stamp the hash into
  the built DB (re-running this script with a future `--room-identity-hash` option, or a small M3 step)
  or use `fallbackToDestructiveMigration`-style handling. The camelCase columns here are chosen to match
  Room's default column naming so the entities line up.

## 11. Risks / open items

- **Scryfall API/schema drift:** field names (`oracle_id`, `oracle_text`, `card_faces`, `comment`) are
  stable, but the script should fail loudly if the bulk-data endpoint shape changes.
- **DB size / build time:** full oracle set (~30k cards) + rulings — expect tens of MB and a run of a
  minute or two; `VACUUM` compacts it.
- **Rate limits:** only 3 network calls (metadata + 2 downloads); include a `User-Agent` and be polite.
